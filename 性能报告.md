## 测试方法

基准工具：`mydb-bench`（`crates/mydb-bench`）

对比参考：MySQL 8.0.46（历史数据，`innodb_flush_log_at_trx_commit=1`）

| 场景 | actors | tables | transaction_size | writes_per_actor | 说明 |
|------|--------|--------|-----------------|------------------|------|
| 单表写 (fsync-per-commit) | 1 | 1 | 1 | ≥500 | 单连接单行 INSERT，每次 COMMIT 一次 fsync |
| 8 actor / 8表 写 P99 延迟 | 8 | 8 | 10 | 500 | 高并发游戏负载，P99 尾延迟 |
| 8 actor / 8表 Group Commit | 8 | 8 | 1 | 500 | 并发写入自然批量聚合吞吐 |
| 读 P50 延迟 | 1 | 1 | - | 50 reads | 单连接点查主键延迟 |

## 发布前更新流程

```bash
# 1. 启动 MyDB（Docker 或本地）
docker compose up -d --build

# 2. 启动同配置 MySQL 8.0 作为对比基线

# 3. 单表写基准
cargo run -p mydb-bench -- --url mysql://root:root@127.0.0.1:3306 \
  --actors 1 --table-count 1 --transaction-size 1 \
  --writes-per-actor 1000 --reads-per-actor 0

# 4. 8 actor / 8表 并发基准
cargo run -p mydb-bench -- --url mysql://root:root@127.0.0.1:3306 \
  --actors 8 --table-count 8 --transaction-size 10 \
  --writes-per-actor 500 --reads-per-actor 50

# 5. 将结果填入上方「当前版本性能」表格，更新 commit hash 和日期
```

## 设计目标

MyDB 设计目标是**同资源、同持久化级别下达到 MySQL 10x 写性能**，不以关闭持久化换取数字。

关键性能原则：
- 每次提交仅一次 `sync_data()` 顺序 fsync
- 锁内不做序列化/分配
- 自然批量优先，不人为引入延迟
- 热路径零堆分配
